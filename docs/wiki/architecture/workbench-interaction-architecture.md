# Workbench Interaction Architecture

The borehole workbench should behave like a reusable interaction engine, not a fixed page.

## Goals

- Every track should use the same depth mapping.
- Click, hover, right-click, and future drag interactions should produce standardized events.
- Track-specific hit testing should be isolated inside the track module.
- Application behavior should be handled centrally.
- The workbench engine should be reusable in another UI project.

## Core Concepts

The design is intentionally lightweight. It borrows useful concepts from specialized well-log/mining visualization toolkits, but avoids building a large framework before the product needs it.

The minimum abstraction we want is:

```text
TrackContainer
  -> shared scales and interaction dispatch
Track
  -> one visual lane with its own renderer and hit testing
TrackObject
  -> typed geological/log object under the pointer
Renderer
  -> drawing logic for a specific track type
Interaction handler
  -> application behavior for click, hover, context menu, drag
```

This keeps the system ready for complexity while still allowing fast product work.

### Track Container

The Log Widget owns:

- visible track list
- total depth
- shared depth scale
- selected depth crosshair
- centralized interaction handler

### Depth Scale

`createDepthScale()` maps:

- depth to screen Y
- screen Y to depth
- depth to percent
- interval from/to depths to CSS position

All tracks should use this scale instead of implementing their own math.

### Virtual Range And Visible Range

The log has two depth concepts:

- virtual range: the full borehole depth, such as `0m - 604.3m`
- visible range: the current viewport after zoom, such as `150m - 220m`

Tracks render against the visible range, but data lookup and hit testing still operate against the full virtual range.

Curve rendering should include boundary samples just outside the visible range when available. This prevents visual gaps at the top and bottom of a zoomed curve while preserving raw nearest-sample inspection behavior.

Intervals, seams, remarks, RQD, and recovery tracks can filter to objects intersecting the visible range.

### Tick Generation

Depth ticks are generated by a small utility instead of hardcoded marks.

The tick generator uses:

- visible depth range
- pixels per meter
- target pixel spacing
- nice numeric intervals such as 1, 2, 5, 10, 20, 50

This lets the depth track show coarse ticks in full-borehole view and more precise ticks when zooming is added later.

The same pattern can later support:

- curve value ticks
- grid lines
- TVD/deviation tracks
- synchronized cross-track cursors

### Track Frame

`TrackFrame` wraps each track and handles:

- mouse hover
- click
- right-click/context menu
- local mouse coordinates
- mouse depth calculation
- calling the track module's `hitTest`
- emitting a standardized `TrackPointerEvent`

### Track Object

A track object represents what the pointer interacted with:

- `lithology-interval`
- `seam-interval`
- `curve-sample`
- `core-image`
- `remark`
- `depth`
- `empty`

Each track object includes a depth and the relevant domain object.

### Hit Testing

Each track owns its own hit test logic:

- lithology track finds the interval at depth
- seam track finds the seam at depth
- curve track finds the nearest measured sample
- image widget can find image evidence by depth
- future remark track can find grouped events by depth

Hit testing should return a `TrackObject`, not directly mutate UI state.

### Central Interaction Handler

`handleTrackPointerEvent()` decides what happens:

- hover curve sample -> show tooltip
- click interval -> select interval and depth
- click curve sample -> pin selected depth
- right-click any track -> show context menu

Future edit/drag behavior should be added here or delegated from here, not scattered across tracks.

## Extension Rules

When adding a new track:

1. Create a track renderer under `frontend/src/workbench/tracks/<track-name>/`.
2. Wrap it in `TrackFrame`.
3. Use the shared `DepthScale`.
4. Implement `hitTest`.
5. Return a typed `TrackObject`.
6. Let the central interaction handler decide behavior.

Avoid adding a new abstraction unless it helps with one of these:

- consistent depth/value mapping
- reusable hit testing
- configurable rendering
- cross-track interaction
- performance for dense curves or many objects
- clean migration to another UI project

When adding a new interaction:

1. Add a standardized event type if needed.
2. Add behavior in the central handler.
3. Keep track renderers focused on drawing and hit testing.

## Why This Matters

This design lets us add:

- editable interval boundaries
- drag-to-split intervals
- curve point inspection
- cross-track cursor
- context menus
- AI suggestion markers
- image/remark evidence objects
- future deviation tracks

without rewriting the Log Widget each time.
